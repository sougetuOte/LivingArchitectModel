## 6. 知見の定着

本 retro の最大の発見は **「Stage 2 反省点を Stage 3 委譲 prompt にガードレール化 → L2 報告品質 50% → 5%」** の実証。
L2 (Sonnet / tdd-developer) は十分な能力を持つが、Bash 権限制約・緩和判断境界・計測法・既存テスト影響評価といった
「手順面の前提」を明示しないと報告精度が落ちる。

→ 次回 Wave 委譲時は **委譲 prompt の冒頭にガードレール 4 点をテンプレ挿入** することを標準化する。

---

## 7. ガードレール 4 点（Stage 3 で実証）

```
1. Bash 制限前提: 権限のない Bash コマンドは試行せず、L1 に依頼すること
2. 緩和事前承認必須: 既存テスト・規約からの緩和は実施前に L1 へ承認依頼
3. JS 行数計測法明示: 4 区分（実装行 / コメント行 / 空行 / 合計）で計測し、SHOULD 100-150 範囲と突合
4. 既存テスト影響事前分析: 改修対象の波及範囲を事前に列挙し、破損予測を委譲時に共有
```

これらは次 Wave 委譲時の prompt テンプレに組み込む候補。
