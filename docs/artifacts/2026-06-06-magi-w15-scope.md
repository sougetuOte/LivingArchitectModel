## 各 Atom の結論

### A1: symlink 展開の実装方針

**確定**: 前回設計案を採用。
- 絶対パス分岐にのみ `resolve()` を導入（両辺 resolve）
- `resolve(strict=False)` で非存在 Write パスも親まで解決（Write 経路の回帰防止）
- out_of_root マーカーの表示は生 `file_path` を維持（既存テスト互換・ログ可読性）
- 相対分岐は不変
- **追加条件**: `resolve()` 失敗（循環 symlink の OSError / 深い再帰の RuntimeError）時のフォールバック（`resolved = p`）で **WARNING ログを残す**（握りつぶし禁止の作法。`get_project_root` の LAM_PROJECT_ROOT フォールバックと同様）。フォールバックは out_of_root 側（厳しい側）に倒れるためフェイルセーフ。

### A2: 相対パス分岐に resolve を入れるか

**確定**: 相対分岐は不変（resolve を入れない）。
- 理由は cwd 依存ではなく、「相対パス素通し契約」（pre-tool-use の PM/SE パターンが相対パス前提）への介入が Zero-Regression リスクを増やすこと。
- traversal 対策は A2 ではなく A3 のスコープ判断に委ねる。

### A3: path traversal を W-15 に含めるか

**確定**: 別タスク起票。
- 脅威クラスが別（symlink = 物理実体の偽装 / traversal = 相対パス構文の越境）。
- W-15 の完了条件（symlink 経由が project_root 外と判定される）の膨張を防ぐ。
- 良性 `..`（`docs/../specs` 等の正規 root 内パス）の誤検知設計が別途必要なため、独立タスクが適切。

## Reflection

致命的な見落とし検査:
- セキュリティ: A1 フォールバックは out_of_root（厳しい側）+ WARNING で可視化 → 無効化せず。
- データ損失: なし（判定のみ）。
- 仕様違反: A2 で相対契約保持 → pre-tool-use の PM/SE 照合不変 → Zero-Regression 維持。
- 表示の生パス保持: 判定は resolved で実施、表示のみ入力パス → 仕様上正しい。

**致命的な見落とし: なし → 結論確定**

## 統合結論

W-15 は **symlink 展開に限定**して BUILDING 着手。前回設計案 + フォールバック WARNING ログを採用。path traversal は別タスクに分離。
