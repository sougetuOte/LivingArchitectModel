### 1.2 詳細

#### NFR-1: 月額LLM利用費 $100以内（Claude Code Max x5）

- **定義**: 月額LLM利用費を $100以内に収める（NFR §6.1、ADR-0001 関連）
- **集計実装**: 無。自動集計スクリプトなし。ADR-0001 で「Opus 枠をメインセッションに温存」の方針のみ定義
- **出力先**: 無（ユーザーが Settings > Usage で手動確認する運用）
- **使用箇所**: `docs/adr/0001-model-routing-strategy.md`（関連要件として参照）、`docs/artifacts/feat-v4.0.0-immune-system.md:783`（NFR-001 として言及）
- **最終更新**: `docs/adr/0001-model-routing-strategy.md` - コミット 8a42c18（2026-06-13 ごろ）
- **使用頻度**: low（モニタリングは手動のみ、自動集計なし）
- **失敗時影響**: high（予算超過は運用継続に直結）
- **マトリクス分類**: 温存（fail-safe）— 低頻度・高影響の quadrant
- **削減プラン**: 測定インフラ構築はv5スコープとし、現定義は維持する。`goal-driven-orchestration` の NFR-1（コスト可観測性）と統合を検討（要追加調査）

#### NFR-2: PG級タスクのモデル = Sonnet / Haiku（コスト削減）

- **定義**: PG級処理は軽量モデルを使用する
- **集計実装**: 有。`.claude/agents/` のフロントマターに `model: haiku`（task-decomposer, test-runner）および `model: sonnet`（それ以外）が設定済み
- **出力先**: `.claude/agents/task-decomposer.md`, `.claude/agents/test-runner.md`
- **使用箇所**: 全 8 エージェントのフロントマター（grep ヒット: `model: haiku` 2件、`model: sonnet` 6件）
- **最終更新**: 2026-06-13（直近コミット ec84f13 以降、agent 定義は変更あり）
- **使用頻度**: high（全エージェント起動時に適用）
- **失敗時影響**: medium（コスト増加・レート消費。機能は停止しない）
- **マトリクス分類**: 温存
- **削減プラン**: 現状維持。hooks（command型, PG）はモデル不使用で設計通り

#### NFR-3: PM級タスクのモデル = Opus（精度優先）

- **定義**: PM級処理は高精度モデル（Opus）を使用する
- **集計実装**: 部分。エージェントは全て `model: sonnet`。Opus は「Fable=判断・PM整理のみ」として CLAUDE.md に人間体制レベルで運用（MEMORY.md 参照）
- **出力先**: `.claude/agents/*.md`（sonnet で統一）
- **使用箇所**: ADR-0001 §決定（B: command→Haiku→Sonnet の3層）で Opus をメインに温存する設計が確定済み
- **最終更新**: エージェント定義 2026-06-13、ADR-0001 2026-03-08
- **使用頻度**: medium（PM 級判断は人間＋Opus が手動で実施）
- **失敗時影響**: medium（Sonnet で代用中、精度低下リスク）
- **マトリクス分類**: 共通化検討
- **削減プラン**: CLAUDE.md の「3層作業体制（Fable=Opus/判断）」と整合させ、仕様書記述を「Opus は人間起動時に使用（自動フック・エージェントは Sonnet 上限）」に更新（SE 級・PM 判断後）

### 2.3 fat 化判定（軸 C）

- **現状 fat の有無**: 無し（2 エントリ・19 行は最小規模。実害は発生していない）
- **放置時の膨張予測**:
  - goal-driven が実運用に入ると、タスク 1 件ごとに 1 エントリ（約 9〜10 行）が追加される
  - fail→pass 遷移がない小タスクは全て「未検証・定型文」エントリとして蓄積される
  - 例: 月 20 タスク実行 × 6 ヶ月 = 120 エントリ（約 1,200 行）。そのうち未検証が大多数を占める可能性が高い
  - `task_id` 重複防止はあるが、内容類似の別 `task_id` エントリは無制限に蓄積する
- **重複スキップ機構の優先度**: 低
  - 現実装の `task_id` idempotency は適切に機能しており、明示的な「重複」（同一 task_id の二重追記）は発生しない
  - 「意味的な重複」（異なる task_id で同じ内容）に対しては無効だが、これは現状件数では実害なし
- **新規性チェック機構の優先度**: 中
  - 未検証エントリ（「（自動抽出・要人間確認）」定型文）が lessons.md の大部分を占める可能性が高い
  - これは /retro Step 4 での人間精査を前提とした設計であり、意図的な動作（design §13 MUST）
  - ただし未検証エントリが大量蓄積すると /retro の精査コストが上がるため、中優先度での対処が望ましい
- **マトリクス分類**: 実装追加（fat 予防）
  - 削除候補や温存ではなく、「現状は無害だが将来的に低価値エントリが膨張する構造的問題」
  - 重複スキップ強化は低優先・現状維持。新規性チェックの部分的追加を中優先で検討

### 4.3 廃止 vs 統合 判定（軸 C）

#### C1: Reflection が機能している証拠

**証拠: 無し（初回 Reflection での変更ゼロ / 7 件中 7 件が「なし → 確定」）**

- 7 件の記録済み Reflection のうち、初回 Reflection で結論が変わったものは 0 件
- 唯一の実質的な「変更」は ADR-0005 の Reflection 追補（#4）だが、これは adversarial な外部検証（DW ブラインド検証）によるものであり、MAGI Reflection 機構そのものではない
- SESSION_STATE.md §問題 の記述「MAGI Reflection が機能していない疑い: 入力同一のため 2 回目で結論が変わらない（本セッション議論）」が実機データで裏付けられた

#### C2: 機能していない場合の根因分析

**根因 1: 入力同一問題（主因）**

Reflection は Step 3 (CASPAR 収束) の直後に、同一文脈・同一 LLM が実行する。Step 3 で CASPAR が既に「最悪の場合」を考慮して収束しているため、Reflection ステップで追加の情報なしに同じ入力を再処理しても同じ結論が出る。

```
Step 3 (CASPAR): 両視点を統合 → 結論
Step 4 (Reflection): 同一文脈で結論を検証 → 同じ結論（入力不変のため）
```

**根因 2: バイアス方向性の同一性**

CASPAR が収束の際に既に「致命的見落としがないか」を考慮しているという前提がある。Reflection が「同一バイアス構造の 2 回目チェック」になる。

**根因 3: 閾値設定（致命的のみ）**

Reflection の修正条件が「致命的な見落とし（セキュリティ、データ損失、仕様違反）」に限定されている。Step 3 で安全側に倒した結論に対して、致命的な見落としが存在するケースは構造的に希少。

**根因 4: 記録上の形骸化**

「致命的な見落とし: なし → 結論確定」がテンプレ文言として定着しており、形式的に記述されている可能性がある。実際に検証を行ったかどうかが外部から検証できない。

#### C3: v5 構想 ② gabriel 統合との整合性

SESSION_STATE.md の v5 構想 ②「MAGI v2」設計:

```
gabriel（出題・決断者）
  ├─ MELCHIOR / BALTHASAR / CASPAR（純調停化）
  └─ サブ調査者（裏取り・根拠調査）
Reflection → 廃止 or gabriel に統合
  推奨案 C: gabriel 決断前の adversarial probe 工程に吸収
  副次案 B: 真の再評価化（逆張り検証）
```

整合性評価:
- プラン B（gabriel 統合・adversarial probe 化）は ADR-0005 Reflection 追補（#4）が実証したパターン（独立文脈での検証が有効）と整合する
- 現行 Reflection の「同一文脈での再確認」を廃止し、gabriel という独立エージェントによる「adversarial probe」に置き換えることで、根因 1（入力同一問題）と根因 2（バイアス同一性）が解消される
- 現在の Reflection コストが小さい（5-10% ボリューム）ため、廃止による直接コスト削減効果は限定的。主な効果は「形骸化したステップの除去」と「将来の gabriel 統合への地ならし」

#### C4: 3 つの削減プラン評価

| プラン | 概要 | メリット | デメリット | コスト | リスク |
|--------|------|---------|-----------|--------|--------|
| A: 完全廃止 | Reflection を全ステップから削除 | 形骸ステップ除去・即時実施可 | adversarial 検証ゼロ化・MAR 研究根拠の破棄 | 低（ルール・SKILL.md 更新のみ SE 級） | 低（機能していない機構を削除）|
| B: gabriel 統合 | gabriel 導入 + adversarial probe として吸収 | 根因を解消・実証済みパターン（ADR-0005 追補）と整合 | gabriel 導入が PM 級・v5 MAGI v2 設計と同時実施が必要 | 高（新エージェント設計・MAGI 全体再設計） | 中（gabriel の詳細設計が未確定） |

- Reflection 記録件数: 7 件（2 件は記録なし）
- Reflection で結論変更（初回）: **0 件 / 0%**
- 「致命的な見落とし: なし」テンプレ出現: **7 件 / 100%**

**注目所見**:

1. **「致命的な見落とし: なし → 結論確定」が 7 件中 7 件（100%）**: 記録された全 Reflection が同一テンプレ出力。Step 3 (CASPAR) 直後の同一文脈再処理であり、入力不変のため構造的に結論変更が起きにくい設計になっている

2. **唯一の実質的 catch は MAGI Reflection ではなく外部 adversarial 検証（ADR-0005）**: DW ブラインド検証（別セッション・独立文脈）が Reflection 追補で 3 catch を発見。このパターンこそがプラン B (gabriel adversarial probe) の設計根拠になる

3. **廃止コストは小（PM 級指摘止め）、統合コストは大（gabriel 設計全体）**: プラン A は即時実施可能だが adversarial 検証ゼロ化のリスクあり。プラン B は効果が高いが v5 全体設計との同時実施が必要

4. **「Reflection の Reflection は禁止」という設計制約が外部 adversarial 検証へのニーズを証明している**: ADR-0005 が Reflection 追補（別セッション）を必要とした理由は、Reflection の再実行が禁止されているためでもある。この制約自体が「1 回の同一文脈再確認では不十分」を認めている

5. **集計の限界**: 記録されていない MAGI 合議（daily 言及のみ）が少なくとも 2 件存在する。独立アンカーファイルが作成される合議と daily log のみに記録される合議の区別基準が定義されていない。完全な集計は不可能であることを明記する
