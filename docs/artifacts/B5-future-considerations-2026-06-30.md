## 論点 1: Claude Code ロードマップ変化と LAM 設計思想のギャップ

### 観測 (2026-06-30 Boris Cherny @bcherny 発表 / 一次ソース未確認)

- サブエージェントのバックグラウンド実行が既定化
- 許可リクエストがメインエージェントに上位転送される
- ↓ + Enter によるサブエージェントへの zoom-in
- 別タブ展開機能 (近日公開)
- `claude -p` でも箱出しサポート
- 「新モデル」への言及あり

### 仮説

Claude Code は「同期ペアプロ」から「**非同期ディスパッチャ + 並列ワーカー**」モデルへ移行している。方向性として **エージェント自律性を高める** 路線。LAM の「人間の制御権を保つ」方向 (セーブ/ロード、MAGI、フルレビュー、承認ゲート) と方向性が逆。

### 派生論点 (B-5 retro で MAGI 合議対象)

1. **3.5 層チーム階層は subagent nesting に吸収される可能性**。但し「役割固定・審議強制・承認締め」の規律は CC プリミティブからは出てこない → ここが LAM の核心
2. **L2 実行ワーカー層の役割縮小**。CC 側が太るほど LAM 独自 L2 は冗長化
3. **LAM の売りを機能ベースで考えるとじり貧**だが、「人間の制御権を保つフレームワーク」として再定義すると、CC が自律性方向に進むほど相対的価値が上がる可能性

### 未確定事項 (B-5 retro までに裏取り)

- subagent nesting 時のコンテキスト管理の安定性 (Cherny 自身が `/bug` 報告を求めているレベル)
- 新モデル発表タイミング (Mythos 規制下で即 GA とは限らない)
- マルチプロバイダ抽象化レイヤーの判断 (アダプタ追従コスト vs ORM 的失敗リスク)
- **一次ソース裏取り (upstream-first)**: Boris Cherny の発表本文・公式 changelog・docs 上の記述

### B-5 retro での扱い

B-5 全 Wave 完了後の retro の Try/Action 欄で MAGI 合議の議題として取り上げる。AoT 分解で派生論点 1〜3 を Atom 化し、ADR-0008 とは別軌道の **ADR-0009 (LAM 戦略方針の再定義 / 暫定名)** 起票要否を判断する。
