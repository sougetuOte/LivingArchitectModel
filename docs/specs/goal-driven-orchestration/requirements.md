## 0. 前提条件・運用要件

### 課金上限の事前設定（人間側の必須作業）

本スキルが設定する bound（FR-4）はモデルへの指示レベルで機能するが、プラットフォームの
課金動作は LAM の bound では制御できない外部要因である。

- サブスクプランの使用量超過時、使用クレジットへ自動的に課金が溢れる設定がデフォルトになっている
  場合がある。この場合、スキル起動前に bound を設定しても API 課金は発生し得る
- **本スキルを使用する前に、Settings > Usage でハードキャップを確認・設定しなければならない（MUST）**
- ハードキャップの設定は人間（PM）の責任であり、スキル実装では代替できない

### 1M コンテキストと自動コンパクト

- 1M コンテキストセッション使用時、クレジット不足時に自動コンパクトが発生する（恒久スタック解消済み）
- 長期セッションでは自動コンパクトの発動によりコンテキストが圧縮される場合があることを前提として
  設計すること。コンパクト後も rubric.md が全層から参照可能な状態を維持しなければならない（MUST）

## 1. Problem Statement

LAM（Living Architect Model）の BUILDING フェーズは現在、PMが直接上位モデルへタスクを投入する
単層構成が主体である。この構成では以下の問題が生じる:

1. **コスト非効率**: 上位モデルが実装・テスト・ログ読取まで担い、トークン消費倍率の高いモデルに
   最もトークンを食う役割を担わせる逆転が起きる
2. **自走性の欠如**: モデルが「完了した」と自己申告しても独立検証が存在せず、品質保証が人間依存
3. **スケール欠如**: タスク規模に応じた構成切り替え機構がなく、小タスクでも重い構成を使う

本スキルはゴール（rubric）を環境側に置き、「実行→評価→修正」の自己修正ループを構成することで
上記 3 点を解消する。設計思想の根拠は参考資料 [R1] の Lance Martin による知見に基づく。

---

## 2. Goals

1. タスク規模（小/中/大）に応じたモデル構成を自動選択し、BUILDING の日常主力スキルとして機能する
2. 上位指揮者モデルの仕事を「rubric 設計・最終検収」に限定し、実行コストをワーカー側に抑制する
3. 合否判定を独立したコンテキストの grader に委ね、作業者の自己申告を合格条件にしない
4. ループの上限（ターン数・時間・トークン）を必ず設定し、暴走と過剰課金を防ぐ
5. タスク終了時に失敗と修正を「検証済みの一般則」として LAM メモリ機構へ蒸留する

---

## 3. Non-Goals（今回の対象外）

- CMA（Claude Managed Agents）への移植（設計は CMA の Outcomes/memory と概念互換に保つが実装はローカル Claude Code 前提）
- Dynamic Workflows を用いた大規模並列ファンアウトの実装（特大タスクルートは将来の別フェーズ）
- PLANNING フェーズや AUDITING フェーズの処理の置き換え（既存 lam-orchestrate・full-review を置き換えない）

---

## 3b. 設計原則

1. **プラットフォーム依存は薄く・差し替え可能に保つ**: `/goal`・workflow・hooks 等のプラットフォーム
   機能を利用する箇所は LAM 固有層（rubric 生成・メモリ蒸留・コスト集計・full-review 連携）と
   明確に分離し、プラットフォーム仕様変更時に LAM 全体が壊れない構造にしなければならない（MUST）
2. **LLM 呼び出しは判断箇所のみ**: 難易度判定・rubric 生成・最終検収など LLM の判断が必要な箇所
   のみをモデル呼び出しとし、フロー遷移・分岐・エスカレーションなどの決定的処理は
   モデル呼び出しにしてはならない（SHOULD NOT）

## 4. モデル編成（三層デフォルト）

| 層 | 役割 | 担当モデル | 職責の制限 |
|----|------|-----------|-----------|
| L1 指揮者 | タスク分解・rubric生成・最終検収 | 最上位モデル（設定値で指定） | コード・ログを直接読まない。構造化報告のみ読む |
| L2 班長 | 工程采配・L3分配・報告集約 | Sonnet 4.6（差し戻し頻発時は Opus 4.8 へ昇格） | 大タスクのみ介在。中/小タスクでは省略 |
| L3 実行者 | 実装・テスト | Sonnet 4.6 / Haiku 4.5 | Task ツールを持たない（R7） |
| grader | rubric 照合・合否判定 | Haiku 4.5（判断が重い場合は Sonnet） | 作業者と別コンテキストで実行 |
| 重実装支援 | アーキテクチャ判断を伴う難所 | Opus 4.8 | L2 の判断で必要時のみ投入 |

**根拠**: 最上位モデルは API 単価が高く（入力 $10/MTok・出力 $50/MTok）、
サブスクプランでは 2 倍消費としてカウントされる。
トークンを大量消費する実装・テスト・ログ読取を L1 に担当させると
タスク完了あたりの総コストが著しく増大するため、L1 の職責を判断・設計のみに限定する。
**目標値**: L1（指揮者）のトークン消費を全体の 20% 以下に抑える（実測に基づく初期目標値。NFR-1 の
コスト集計結果を受けて PM が調整する）。

## 5. 機能要件（FR）

### FR-1: rubric ファースト（原本 R1 対応）

- L1 指揮者は、中タスク・大タスクの着手前に `rubric.md` を生成しなければならない（MUST）
- rubric.md はチェック可能な項目の列挙とし、各項目に検証方法（実行コマンド・テスト・grader 判定のいずれか）を併記しなければならない（MUST）
- rubric.md は全層が直接参照できる共有ファイルとして配置し、伝言・要約経由での受け渡しを行ってはならない（MUST NOT）

### FR-2: 独立 grader（原本 R2 対応）

**用語定義**: 本書で「grader」とは、作業者とは別のコンテキストで rubric を照合し合否と理由を返す
評価エージェントを指す。「軽量 grader」は grader の標準構成（原則 Haiku、判断が重い場合は Sonnet）
を意味し、全箇所でこの定義を適用する。

- 合否判定は、作業者とは別のコンテキストで動作する grader が行わなければならない（MUST）
- 作業者自身の「完了した」という報告を合格条件として使用してはならない（MUST NOT）
- grader は原則 Haiku を使用し、rubric 各項目の合否と理由を返さなければならない（MUST）

### FR-3: 構造化報告（原本 R3 対応）

- 下層から上層への報告は以下の定型フォーマット **のみ** で行わなければならない（MUST）:
  `{ 変更点 / テスト結果 / 未解決事項 / 次の提案 }`
- 上層に全コンテキスト・全 diff を直接渡してはならない（MUST NOT）

### FR-4: bound（多層対応）（原本 R4 対応）

- 各実行ループに上限（ターン数または時間またはトークン数のいずれか）を設定しなければならない（MUST）
- `/goal` を内側ループに使用する場合、条件文に打ち切り句（例: `or stop after 15 turns`）を必ず含めなければならない（MUST）。省略は禁止（MUST NOT）
- グローバル bound を必ず併設しなければならない（MUST）。各層の max_iterations の積が実質上限にならないよう、ツリー全体を総トークン数または総時間で縛る
- bound 超過時は人間（PM）にエスカレーションしなければならない（MUST）

### FR-5: メモリの蒸留（原本 R5 対応）

- タスク終了時（合格・エスカレーション問わず）、失敗と修正の記録を検証済みの一般則に蒸留し、LAM の既存メモリ機構（`.claude/agent-memory/` または専用 lessons ファイル）へ追記しなければならない（MUST）
- 未検証の推測は「未検証」と明記した上で記録すること（MUST）

### FR-6: 三段階ルート（原本 R6 対応）

- L1 指揮者は難易度判定を最初に行い、以下のルートを選択しなければならない（MUST）:
  - **小タスク**（以下の条件をすべて満たす: rubric 項目数 3 以下 AND 未解決質問 0 AND 工程数 1〜2 以内）:
    L3 直行（Sonnet + 軽量 grader。L1 は関与しない）
    ※ 初期値。設計フェーズで実測に基づき調整可
  - **中タスク**: L1 → L3 の二層
  - **大タスク**（工程数が 3 以上または並列分解が必要）: L1 → L2 → L3 の三層
- 小タスクに指揮者を介在させてはならない（MUST NOT）（過剰委任によるコスト逆転の防止）

### FR-7: 最小権限（原本 R7 対応）

- L3（実行者・grader）のエージェント定義から Task ツールを除外しなければならない（MUST）
- 末端エージェントがサブエージェントを自律 spawn することを禁止する（MUST NOT）

### FR-8: Dynamic Workflows 除外（原本 §7.3 対応）

- 小/中/大いずれのルートのデフォルト経路にも Dynamic Workflows を組み込んではならない（MUST NOT）
- effort 設定が意図せず `ultracode`（xhigh）になり Dynamic Workflows が自動起動しないよう、
  effort 設定を明示的に文書化しなければならない（MUST）
- 将来「特大」タスクルートで Dynamic Workflows を使用する場合も、起動には人間（PM）の明示承認
  ゲートを設けなければならない（MUST）。自動起動・暗黙起動は禁止（MUST NOT）（OQ-4 参照）

### FR-9: full-review 連携（原本 §8.2 対応）

- full-review の改造に着手する前に、現仕様分析・互換性影響・工数見積もりを文書として PM へ
  提示しなければならない（MUST）。承認なしに改造に着手してはならない（MUST NOT）
- フロー [8] 納品前検証として full-review をサブエージェントとして呼び出せるインターフェースを定義しなければならない（MUST）
- full-review はトップレベルコマンドとしての既存用途を温存しなければならない（MUST）
- full-review が rubric ファイルを入力として受け取り、検証基準を共有できる形式を定義しなければならない（MUST）

### FR-10: lam-orchestrate 接続（原本 §8.3 対応）

- lam-orchestrate の出力（設計文書・rubric 素案）を本スキルのフロー [2] の入力として受け取れる受け渡し形式を定義しなければならない（MUST）
- lam-orchestrate と本スキルは競合ではなく直列接続（PLANNING → BUILDING の上流・下流）として機能しなければならない（MUST）

## 6. 非機能要件（NFR）

### NFR-1: コスト可観測性（原本 S4 対応）

- タスクごと・モデルごと・層ごとのトークン消費量を集計し、サマリを出力するべきである（SHOULD）
- 特に L1（最上位モデル）の消費比率を監視可能にするべきである（SHOULD）
- 監視目標: L1 のトークン消費比率が全体の 20% 以下であることを確認できること（初期目標値。
  タスク完了あたりの総コスト比較で PM が調整する。§4 モデル編成根拠参照）

### NFR-2: /goal 優先（原本 S1 対応）

- 内側ループには `/goal` を第一候補として使用するべきである（SHOULD）
- 自前 grader が必要になるのは rubric が `/goal` の条件記述で表現できない場合に限り、
  その場合は理由を設計文書に記録しなければならない（MUST）

### NFR-3: grader ログ保持（原本 S2 対応）

- grader 判定のログ（合否理由）を残し、rubric 自体の改善に活用できるようにするべきである（SHOULD）

### NFR-4: LAM フェーズ整合（原本 S3 対応）

- 本スキルは LAM の既存 PLANNING / BUILDING / AUDITING フェーズと整合するべきである（SHOULD）

### NFR-5: ネスト深さの外部化（原本 §3.2 対応）

- サブエージェントのネスト深さ上限（現在暫定値 5）をハードコードしてはならない（MUST NOT）
- 深さ上限は設定値として外部化し、変更時にコード修正を不要にしなければならない（MUST）

### NFR-6: フォールバック（原本 S5 対応）

- ネスト機能が利用不可になった場合の三層→二層への自動退避を検討するべきである（SHOULD）
- フォールバック有無・方式は設計文書に記録しなければならない（MUST）

### NFR-7: 既存 LAM との競合時の先行提示義務（原本 §12 対応）

- フェーズ遷移ルール・承認ゲート・hooks・MAGI 審議と競合が判明した場合は、作業に着手する前に
  変更案を PM へ提示しなければならない（MUST）
- 実装担当が不明点・矛盾を発見した場合は、着手前に質問として列挙しなければならない（MUST）

### NFR-8: 成果物自体の独立 grader 検収（原本 §12 対応）

- 本スキルの実装成果物（SKILL.md 等）は、実装担当者の自己申告完了を合格条件としてはならない（MUST NOT）
- 成果物の検収も独立した grader（または PM）による確認を経なければならない（MUST）
  （FR-2 の独立 grader 原則は本スキル自身の開発プロセスにも適用される）

## 7. 処理フロー（標準）

```
人間: タスク投入（スキル起動）
  ↓
[1] L1 指揮者: タスク分解、難易度判定（小/中/大）
      ├ 小 → L3 直行（Sonnet + 軽量 grader、L1 は以降関与しない）
      └ 中/大 ↓
[2] L1 指揮者: rubric.md 生成（FR-1）＋ グローバル bound 設定（FR-4）
  ↓
[3] 〔大タスクのみ〕L2 班長: 工程分解、L3 への分配
  ↓
[4] L3 実行者: 実装ループ
      /goal による自走（打ち切り句付き・FR-4）→ 構造化報告（FR-3）
  ↓
[5] grader: rubric 照合、合否と理由を返す（FR-2）
      ├ 不合格 → [4] へ差し戻し（bound 到達まで）
      └ 合格 ↓
[6] 〔大タスクのみ〕L2 班長: 工程間の整合確認、報告集約
  ↓
[7] L1 指揮者: 最終検収、メモリ蒸留（FR-5）
  ↓
[8] 納品前検証: full-review 実行（FR-9。重い検証はここで一度だけ）
  ↓
[9] 人間: 受領（rubric 逸脱・価値判断が必要な点は明示報告）
```

## 8. 受け入れ条件

### 基本動作

- AC-1: SKILL.md（またはプラグイン構成ファイル）が存在し、LAM から呼び出せること
- AC-2: FR-1〜FR-10 がすべて実装またはテンプレートとして具現化されていること
- AC-3: サンプルタスク 1 件で、フロー [1]〜[9] が一巡するデモログが取得できること（中タスク経路で可）
- AC-4: 三段階ルート（小/中/大）の判定がデモログで確認できること

### 検証の独立性と安全

- AC-5: grader が作業者と別コンテキストで動作していることがログで確認できること（FR-2）
- AC-6: L3 エージェント定義に Task ツールが含まれていないことが確認できること（FR-7）
- AC-7: `/goal` 使用箇所すべての条件文に打ち切り句が含まれていること（FR-4）
- AC-8: グローバル bound（総トークンまたは総時間）が三層全体に効くことがテストで確認できること（FR-4）
  ※ 具体的なテスト実装方式は OQ-2（決定的骨格の実装方式）の選定後に確定する
- AC-9: bound 超過テスト（意図的に達成不可能な rubric を与える）でエスカレーションが発火すること（FR-4）
  ※ 同上、OQ-2 選定後に確定する
- AC-10: Dynamic Workflows がデフォルト経路で起動しないことが確認できること（FR-8）

### コストと運用

- AC-11: モデル別・層別トークン消費サマリが出力されること（NFR-1）
- AC-12: `/goal` のサブエージェント内動作可否の検証結果が設計文書に記載されていること（OQ-1）
- AC-13: 決定的骨格の実装方式（workflow/hooks/自前スクリプト）の比較検討と選定理由が記載されていること（OQ-2）

### 再編関連

- AC-14: full-review の現仕様分析・改造範囲・互換性影響・工数見積もりが文書として PM へ提出され、
  承認されていること（FR-9）
- AC-15: lam-orchestrate → 新スキルへの成果物受け渡し形式が定義されていること（FR-10）

## 9. Open Questions / 検証タスク

### OQ-1: `/goal` のサブエージェント内動作可否（未裏取り・要検証）

**状況**: `/goal` がサブエージェント（L3）内で機能するか、メインセッション専用かが未確認。
**影響**: 検証結果次第で L3 のループ実装方式（`/goal` 流用 vs 自前ループ）が変わる。
**アクション**: 実装着手前に Claude Code の挙動を実測し、設計文書に結果を記録すること。
**注記**: 未裏取りのプラットフォーム仕様のため、設計書には「未確認（要裏取り）」と明記すること（upstream-first 原則）。
**参考**: [R5] `/goal` 公式ドキュメント

### OQ-2: 決定的骨格の実装方式（比較検討が必要）

**状況**: フロー [1]〜[9] の遷移・三段階ルート分岐・エスカレーションといった決定的処理の実装候補は
(a) 保存済みワークフロー、(b) hooks、(c) LAM 自前スクリプト/スキル の 3 案が存在する。
**要件**: LLM 判断が要る箇所（難易度判定・rubric 生成・検収）だけをモデル呼び出しにし、
L1 のトークンを手続き管理に浪費させない方式を選ぶこと。
**アクション**: 実装担当が 3 案を比較検討し、選定理由を設計文書に記録してから着手すること。
**注記**: 各候補の実在性・書式は upstream-first 原則に従い公式ドキュメントで裏取りすること。
**参考**: [R6] [R7] [R8]（Claude Code 関連ドキュメント）

### OQ-4: 「特大」タスクルートの第二フェーズ化（PM判断待ち）

**状況**: Dynamic Workflows を用いた大規模並列ファンアウト（大規模移行・コードベース全体監査等）を
第二フェーズとして依頼するかが未定。
**制約（確定）**: 将来このルートを実装する場合も、起動には人間（PM）の明示承認ゲートを設けることが
必須である（FR-8 参照）。自動起動・暗黙起動は禁止。通常セッションより大幅に多くトークンを消費する
ため、実行前に人間が明示的に承認する設計を維持しなければならない。
**アクション**: 本スキル（第一フェーズ）の実運用後に PM が第二フェーズ化を判断する。

### OQ-5: 小タスクルートにおける grader 起動主体と FR-7 の整合（設計フェーズで解決）

**状況**: 小タスクルートは「L3 直行 + 軽量 grader」構成だが、FR-7 は末端エージェントの自律 spawn
を禁止している。小タスクルートで grader を誰が起動するかが未定義のまま設計に進むと、
FR-7 との矛盾が生じる可能性がある。
**影響**: 小タスク grader の起動主体（スキル呼び出し元 / 別途定義した軽量ディスパッチャ等）を
設計フェーズで明確化する必要がある。
**アクション**: 設計フェーズで OQ-2（決定的骨格の実装方式）と合わせて解決し、設計文書に記録すること。
