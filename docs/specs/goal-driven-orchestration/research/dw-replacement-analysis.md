## 2. MAGI 審議

### AoT Decomposition

| Atom | 判断内容 | 依存 |
|:-----|:---------|:-----|
| A1 | DW は FR-4 bound 機構（トークン定義・時間 bound・状態ファイル）を満たせるか | なし |
| A2 | DW は NFR-1（層別コスト可観測性）を満たせるか | なし |
| A3 | ガバナンス適合: FR-8 撤回のコストと正当性 | A1, A2 |
| A4 | スケジュール影響（6/22 クリティカルパス） | A3 |
| A5 | 部分採用の余地（置き換え以外の DW 活用） | A1〜A3 |

### Atom A1: FR-4 bound 機構の技術適合

**[MELCHIOR]**: `budget` はハード制約（超過時 throw）であり、自前 spawn-time チェックより強い。schema 付き grader 呼び出しで design §8 の「grader 失敗時 1 回再試行 → エスカレーション」MUST 相当がネイティブに得られる。resume 機構（S5）は自前ループでは構築コストが高い価値だ。
**[BALTHASAR]**: bound の中核 3 要素がすべて崩れる。①`budget.total` はスクリプトから設定できず（P3）、FR-4 の段階別 bound（小50k/中150k/大400k）をスキルが強制できない。②spent() は output tokens のみで（P4）、requirements/design が定義する `total_tokens`（サブエージェント総消費）と**計測定義が不一致** — 50,871 対 515 という 2 桁の乖離は近似ですらない。③時間 bound は S2 により計測不能。さらに S1 により `gd-session-state.json` が更新されず、第二防衛線（Stop hook）が盲目になる。
**[CASPAR]**: 結論: **不合格**。FR-4 はトークン定義・時間 bound・状態ファイルの 3 点で DW 上に成立しない。MELCHIOR の利点（schema/resume）は実在するが、bound の不成立を補償しない。

### Atom A2: NFR-1 層別コスト可観測性

**[MELCHIOR]**: 直列実行に限れば spent() 差分で層別の近似値は取れる。
**[BALTHASAR]**: S3 により `agent()` は usage を返さない。一方 Plan B の Agent ツールは `subagent_tokens`（総消費）を呼び出しごとに返す（W0-T1 実測で確認済み: 27,353）。output tokens のみの近似値は AC-11「層別コストサマリ」の計測値として要件の意図を満たさない。
**[CASPAR]**: 結論: **不合格**。NFR-1 は Plan B（Agent ツール）が明確に優位。

### Atom A3: ガバナンス適合（FR-8 撤回の正当性）

**[MELCHIOR]**: S4 は新事実だ。「DW は opt-in 常時要求で自動化と相性が悪い」という PLANNING 時の懸念は、スキル指示書経由なら解消されている。決定的スクリプト制御という DW の性格は「決定的骨格」という B-3 の設計思想とむしろ同型であり、将来の置き換え余地は本物だ。
**[BALTHASAR]**: FR-8 撤回は requirements 改訂（PM級）+ design の §10/§12/§14 連鎖改訂であり、PLANNING の部分やり直しになる。そして A1・A2 が不合格である以上、撤回の技術的根拠がそもそもない。第二防衛線の不全（S1）は Zero-Regression Policy に反する後退だ。
**[CASPAR]**: 結論: **B-3 では撤回根拠不足**。S4・schema・resume は「将来の再審議材料」として記録する価値があるが、現時点の置き換えは要件後退を伴う。

### Atom A4: スケジュール影響

**[MELCHIOR]**: （A3 の結論を受け、主張なし）
**[BALTHASAR]**: 仮に置き換えるなら requirements→design→tasks の改訂チェーンで、6/22 まで 10 日の残期間に対しクリティカルパス（W1-T1→W1-T3）と正面衝突する。
**[CASPAR]**: 結論: 置き換えルートは 6/22 期限と**両立しない**。A3 の結論を補強。

### Atom A5: 部分採用の余地

**[MELCHIOR]**: P1（呼び出し時レジストリ解決）は即座に使える果実だ。W2-T1 で作成する 3 エージェント定義は、Agent ツールでは同一セッションでテストできないが、**最小 DW プローブ経由なら同一セッション内でスモークテストできる**。F-1 の実害が消える。
**[BALTHASAR]**: それは FR-8（デフォルト実行経路からの DW 排除）に抵触しないか？ — 検証用プローブは goal-driven スキルの実行経路ではなく開発時のテストハーネスであり、要件の文言上は抵触しない。ただし PM の明示承認を得て tasks.md に検証手段として明記すべきで、なし崩しの DW 常用への入口にしてはならない。
**[CASPAR]**: 結論: **2 点を採用**。①W2-T1 の完了条件に「同一セッション内スモークテストは最小 DW プローブで実施してよい（開発時テストハーネス用途・FR-8 のデフォルト経路排除とは別物）」を追記提案（PM 承認対象）。②B-4 再審議条件を future-candidates に記録: (a) `budget.total` のプログラム設定手段、(b) `agent()` の usage 返却、(c) スクリプトからの状態ファイル連携 — のうち (a)(b) の解消が確認された時点で再審議。

## 3. 見送り理由と再審議条件（future-candidates 記録）

**見送り理由（段階2: LAM 適合性不合格）**:
- FR-4: `budget.total` のプログラム設定不可（P3）/ 時間 bound 計測不可（S2）/ `gd-session-state.json` 連携不可で第二防衛線が盲目化（S1）

**B-4 以降の再審議条件**（いずれも上流仕様の変更を要するため、再審議時は upstream-first で再裏取りすること）:
1. `budget.total` をスキル/スクリプト側から設定できる手段の出現
2. `agent()` 戻り値での usage（総消費トークン）提供
3. （補助）workflow スクリプトからの状態ファイル連携手段

条件 1・2 の両方が解消された場合、Plan B ループの DW 置き換えを requirements 改訂（FR-8 見直し）として正規に再審議する。
opt-in（S4）・schema（P2）・resume（S5）は再審議時の推進材料として既に確認済み。
