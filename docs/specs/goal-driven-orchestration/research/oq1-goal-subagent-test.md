# OQ-1 実測検証: /goal のサブエージェント内動作確認（W0-T1）

- 検証日: 2026-06-12
- 検証者: Fable 5（メインセッション）+ サブエージェント実機プローブ
- 対応タスク: W0-T1（tasks.md v1.1.0）
- 対応要件: OQ-1 / AC-12 / NFR-2 / design v0.3.0 §8・§18 D1
- **判定: Plan B 確定**（/goal はサブエージェント内で機能しない）

---

## 1. 検証設計

判別可能性を担保するため、以下のプローブを設計した:

- `/goal` 条件文: **決して真にならない条件**（存在しないファイルの存在チェック）+ `or stop after 5 turns` 打ち切り句
- サブエージェントへのタスク: 1 ターン 1 ツール呼び出しで `echo "turn 1"` 〜 `echo "turn 6"` を計 **6 ターン**実行
- 判別ロジック:
  - evaluator が発火する場合 → 5 ターンで打ち切られる（Plan A）
  - evaluator が発火しない場合 → 6 ターン完走する（Plan B）

プローブのプロンプト 1 行目に以下を配置した:

```
/goal the file D:/work7/LivingArchitectModel/.claude/tmp/oq1-never-exists.txt exists or stop after 5 turns
```

### 使用エージェント

design §8 手順 1 は「L3 相当の minimalist エージェント定義（tools: Bash, Read のみ）を一時作成」と
規定しているが、実測中に**副次的発見 F-1**（後述 §4）により同一セッション内では使用不能と判明。
代替として既存の `error-triager` エージェント（tools: Read, Glob, Grep, Bash — minimalist 構成に最も近い・
Task/Agent ツールなし）を `model: haiku` 指定で使用した。L3 相当の最小権限構成という検証条件は維持されている。

## 2. 観測結果（一次情報）

| 観測項目 | 結果 |
|---------|------|
| 完走ターン数 | **6 / 6**（5 ターンで打ち切られず） |
| `/goal` 行の解釈 | **プレーンテキストとして受領**。スラッシュコマンドとしての展開・解釈の形跡なし |
| 外部介入・条件評価の形跡 | **なし**。Haiku evaluator の発火を示すログ・停止・介入は観測されず |
| サブエージェント消費 | 27,353 tokens / 6 tool uses / 69.3 秒 |

## 3. 機構的裏付け（観測の説明）

N=1 の実測だが、以下の機構的説明が観測と整合する:

1. **スラッシュコマンド展開はメインセッションのユーザー入力レイヤーでのみ行われる**。
   サブエージェントへのプロンプトは Agent ツールの引数（＝ユーザーメッセージ相当のテキスト）として
   渡されるため、`/goal` を含めても展開されない（実測で確認）
2. **サブエージェント自身が /goal を発行する経路が存在しない**。
   サブエージェントの出力はツール呼び出しとテキストのみであり、組み込みスラッシュコマンドを
   発行するインターフェースを持たない（構造的に確定。tools に Skill があってもスキルのみで組み込み機能は対象外）
3. design §8 背景の changelog 記述（v2.1.154: evaluator のサブエージェント実行中の誤発火を修正）は
   「evaluator はメインセッション専用設計」という解釈と整合する

## 4. 副次的発見

### F-1: セッション途中で作成したカスタムエージェント定義は **Agent ツール経由では**使用できない（2026-06-12 改訂）

`.claude/agents/oq1-test-l3.md` を作成直後に Agent ツールで `subagent_type: oq1-test-l3` を
指定したところ「Agent type not found」エラー。Agent ツールのエージェントレジストリは
**セッション開始時のスナップショット**であり、途中作成分は反映されない。

**改訂（W0-T2 実測）**: 一方、Dynamic Workflows の `agent()` に `agentType: oq1-test-l3` を
指定したところ**解決・実行に成功**した（プローブ wf_2a43d366-a44）。

**再改訂（2026-06-12 W2-T1 スモークテスト実測）**: 上記の「DW は呼び出し時に解決」という解釈は
**過剰一般化だった**。W2-T1 で作成した 3 定義（goal-driven-*）を 2 回目の Workflow 呼び出し
（wf_b4581430-c71）で指定したところ全件「not found」。同エラーの利用可能リストには
`oq1-test-l3`（**初回 Workflow 実行前に作成**）が載っており、`goal-driven-*`
（**初回 Workflow 実行後に作成**）のみ欠落していた。よって正しい挙動は:

- Agent ツール: **セッション開始時**のレジストリスナップショット
- DW `agent()`: **初回 Workflow 実行時**のレジストリスナップショット（以後セッション内でキャッシュ）

**B-3 実装への影響（再改訂後）**: 途中作成エージェント定義の同一セッション内テストに
**信頼できる手段は存在しない**（DW プローブはスナップショットタイミング依存で不安定）。
W2-T1 等の動作確認は **静的 pytest（採用済み・確実）+ 新セッションでの実機確認** を正とする。
詳細は `research/dw-replacement-analysis.md` §1.1 P1 の訂正注記を参照。

### F-2: トークン閾値書式の検証（design §18 D1 の C-4 残課題）

Plan B 確定により `/goal` 条件文テンプレート自体を使用しないため、
トークン閾値書式（`stop if total_tokens > N`）の有効性検証は**不要化**した。
bound はすべて Plan B のスクリプトループ側（`gd-session-state.json` + spawn-time チェック）で実装する。

## 5. 限界と留意事項

- N=1 の実測である。ただし §3 の機構的裏付けにより、再測で結果が覆る可能性は低い
- 「5 turns」の turn 単位解釈（サブエージェント内部ターン vs メインセッションターン）には
  不確実性が残るが、`/goal` 行が**コマンドとして展開されなかった**こと自体が主証拠であり、
  turn 解釈の如何にかかわらず Plan A は成立しない
- メインセッション（L1）での `/goal` 動作可否は本検証のスコープ外（OQ-1 はサブエージェント内動作のみを問う）

## 6. NFR-2 対応: /goal を使用しない根拠の記録

NFR-2（/goal 優先原則）に対し、本検証により「`/goal` がサブエージェント（L3）内で
機能しない」ことが実測で確認されたため、**Plan B（SKILL.md スクリプトによる自前制御ループ）を
採用する**。これは「プラットフォームネイティブ機能が利用不能な場合のフォールバック」という
NFR-2 の例外条件に該当する。design §8 にも同旨を記録する（W0-T1 完了条件・PM 承認ゲート対象）。
