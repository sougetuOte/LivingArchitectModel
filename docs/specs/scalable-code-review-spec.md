# Scalable Code Review 要件仕様書

**バージョン**: 1.1
**作成日**: 2026-03-14
**ステータス**: draft
**前提**: Claude Opus 4.6 1M コンテキスト
**関連調査**: `docs/specs/scalable-code-review.md`（調査メモ、本仕様の根拠）

---

## 1. 目的

現行の `/full-review` を拡張し、中規模コードベース（目標 30万行、10万行以上で効率低下は許容）に
対応する Scalable Code Review パイプラインを構築する。

**効率低下の定義**: 10万行を超えた場合に以下の低下を許容する:
- 検出精度の低下（見逃しが増える）
- 実行時間の増加（線形以上にスケール）
- 収束イテレーション数の増加

## 2. 背景

### 2.1 現行の問題（もぐらたたき問題）

v4.4.1 の full-review（約3,000行対象）で以下が顕在化:

- 4 エージェントが独立にファイルを読むため、修正の副作用を見逃す
- 修正後の再スキャンで新規 Issue が発生し、収束に時間がかかる
- エージェント間でコンテキストが共有されていない

### 2.2 規模の壁

トークン推定は 1行 ≒ 3トークン（コメント・空行を含む平均値）で算出。

| 規模 | 行数 | 推定トークン | 1M に対する占有率 |
|------|------|-------------|-----------------|
| 小 | ~5K | ~15K | 1.5% |
| 中小 | ~30K | ~90K | 9% |
| 中 | ~100K | ~300K | 30% |
| 中大（目標） | ~300K | ~900K | 90%（溢れる可能性） |

## 3. 要件

### FR-1: 多言語静的解析パイプライン

Python による事前静的解析パイプラインを提供する。
LLM のコンテキストに全コードを載せる代わりに、機械的チェックを事前実行し、
結果サマリーと問題箇所のみを LLM に渡す。

**初期サポート言語**:

| 言語 | lint | セキュリティ | AST パーサー |
|------|------|------------|-------------|
| Python | ruff | bandit, semgrep (*) | ast (標準), tree-sitter (*) |
| JavaScript/TypeScript | eslint | semgrep (*), npm audit | tree-sitter (*) |
| Rust | clippy | `cargo audit`（パッケージ名: cargo-audit） | tree-sitter (*) |

(*) Phase 1 スコープ外。semgrep は Phase 2 以降で追加予定。tree-sitter は Phase 2（AST チャンキング）で導入。

**拡張性要件**: ユーザーが自力で他言語（Go, Java 等）を追加実装しやすい
プラグイン構造を提供すること。言語固有の処理はインターフェースで抽象化する。
<!-- [仮決定] プラグイン検出方式: .claude/hooks/analyzers/ に *_analyzer.py を配置すれば自動検出（動的 import）。設定ファイル不要で直感的。 -->

**ツール未インストール時の動作**: 静的解析ツールがインストールされていない場合は
**エラーで停止**し、インストールを促すメッセージを表示する。
スキャン未実施のコードがすり抜けるリスクをゼロにする設計。

**複数言語混在プロジェクト**: 検出された全言語の Analyzer を実行し結果を統合する。
Phase 1 では逐次実行、Phase 2 以降で並列化を検討する。
ユーザーは除外言語を設定で指定可能。

### FR-2: AST ベースチャンキング

tree-sitter を用いてコードを構文的に妥当な単位（関数/クラス/モジュール）に
チャンク分割する。チャンク間には「のりしろ」（import 文、クラス定義、型情報）を付与し、
周辺コンテキストの喪失を防ぐ。

<!-- [仮決定] のりしろサイズ: チャンクサイズの 20% 以内（目標チャンク 3,000 トークンの場合、のりしろ最大 600 トークン）。cAST 論文の推奨値と一致。 -->

**tree-sitter 依存**: pip install で入る範囲（`tree-sitter`, `tree-sitter-python` 等）は
許容する。tree-sitter がインストールできない環境では Plan B（AST チャンキング）を
丸ごとスキップし、Plan A（静的解析）のみで動作する。

### FR-3: 並列 Map-Reduce レビュー

チャンクごとに独立した LLM エージェントで並列レビュー（Map）し、
結果を統合・重複排除（Reduce）する。

**並列数**: デフォルト 4、最大 10（設定で調整可能）。
6並列以上でコンテキスト破壊の実績があるため、デフォルトは保守的に設定。

**Reduce フェーズの横断チェック**:
- API 呼び出しと定義の一致（呼び出し側と定義側が別チャンク）
- 型の整合性（引数の型が呼び出し先の期待と一致）
- 命名規則の統一性

### FR-4: 階層的レビュー

ファイル → モジュール → システム の 3 階層で段階的にレビューする。
下位レイヤーの要約（概要カード）を上位に渡し、粒度に応じた問題を検出する。

**モジュールの定義**: パッケージ定義の存在で判定する。
Python は `__init__.py` の存在、JavaScript/TypeScript は `package.json` の存在で
モジュール境界を決定する。パッケージ定義がない場合はディレクトリ単位にフォールバック。

### FR-5: 全体再レビュー原則

修正後の再レビューは**対象全体をゼロベースで実施する**。
修正対象のみの部分再レビューは禁止。これは現行 full-review の
Green State 定義（「スキャンして Issue がゼロ」）と一致する。

**理由**:
- 修正の副作用で新たな問題が別箇所に生じうる
- 部分再レビューでは潜在的不具合を見逃すリスクがある
- 効率は Plan D/E の影響範囲分析で将来的に最適化する

**静的解析のキャッシュ**: 再レビュー時、変更ファイルの静的解析は再実行し、
未変更ファイルの静的解析結果はキャッシュを利用する。
LLM レビューはゼロベースで再実行する。
Phase 1 では `state_manager` にキャッシュ基盤（ファイルハッシュ保存・比較）を実装済みだが、
`run_phase0()` からの呼び出しは Phase 2 以降で有効化する。

### FR-6: コンパクション対策

Plan A（静的解析パイプライン）の段階から以下のコンパクション対策を組み込む:

- Issue リスト、構造メタデータを外部ファイル（JSON/MD）に永続化
- LLM コンテキストには判断に必要な情報のみ載せる
- セッション分割時は SESSION_STATE.md で引き継ぎ

依存グラフ、契約カードは Phase 4（Plan D）で追加。Phase 1 では
Issue リスト + AST マップの永続化のみ実装する。

### FR-7: 将来拡張（Plan D: 依存グラフ駆動）

将来的に、依存グラフを構築しトポロジカル順にレビューする手法を導入する。
下流への影響を追跡しながら進み、修正の副作用を最小化する。

**主要コンポーネント**:
- 依存グラフ構築（tree-sitter + pydeps/pyan）
- トポロジカルソートによる処理順序決定
- 強連結成分（循環依存）の検出・マーク
- PageRank による重要度ランキング
- 「契約カード」（公開 API、前提条件、保証）の記録と参照

### FR-8: 将来拡張（Plan E: ハイブリッドパイプライン）

Plan A〜D の長所を統合した最終形態。

```
Stage 1: 静的事前分析（Plan A）
Stage 2: チャンク分割 + 並列レビュー（Plan B + D のグラフ活用）
Stage 3: 階層的統合（Plan C）
Stage 4: 修正 + 全体再レビュー（FR-5 に従う）
Stage 5: Green State 判定
```

## 4. 非機能要件

### NFR-1: 段階的導入

Plan A → B → C の順で段階的に導入する。
各段階で効果を検証し、次段階の着手判断を行う。
詳細は Section 5（実装スコープ）を参照。

### NFR-2: 既存 full-review との互換

小規模プロジェクト（〜10K行）では現行の full-review がそのまま動作すること。
Scalable パイプラインは以下の閾値で有効化する:

| 行数 | 動作 |
|------|------|
| ~10K | 現行 full-review（変更なし） |
| 10K-30K | Scalable パイプラインの使用を提案（ユーザーが選択） |
| 30K~ | Scalable パイプラインを自動有効化 |

### NFR-3: 外部ツール依存の最小化

Python 標準ライブラリ + pip でインストール可能なツールのみ使用する。
Docker や外部サービス（VectorDB 等）への依存は避ける。
tree-sitter のコンパイル済みグラマー（`pip install tree-sitter-python` 等）は許容する。

### NFR-4: Lost in the Middle 対策

1M コンテキストでも中間部分の精度低下（Liu et al., TACL 2024）が発生する。
重要な情報はコンテキストの先頭と末尾に配置する設計を採用する。

### NFR-5: 実行時間の目安

| 規模 | Plan A のみ | Plan A+B+C |
|------|-----------|-----------|
| ~30K | 5分以内 | 15分以内 |
| ~100K | 15分以内 | 45分以内 |
| ~300K | 30分以内 | 120分以内 |

上記は目安であり、ハード制限ではない。大幅に超過する場合はユーザーに
進捗状況を通知し、継続・停止を選択できるようにする。

## 5. 実装スコープ

### Phase 1（直近）: Plan A — 静的解析パイプライン

- Python 静的解析スクリプト群の構築
- 多言語プラグインインターフェース
- コンパクション対策（外部永続化）
- full-review Phase 0 への組み込み

### Phase 2（中期）: Plan B — AST チャンキング

- tree-sitter ベースのチャンキングエンジン
- のりしろ（overlap）設計
- Map-Reduce オーケストレーション

### Phase 3（中期）: Plan C — 階層的レビュー

- 3 レイヤー設計（ファイル → モジュール → システム）
- 概要カード生成
- 階層間の情報伝達

### Phase 4（長期）: Plan D — 依存グラフ駆動

- 依存グラフ構築エンジン
- トポロジカルソート + 契約カード
- 変更影響分析

### Phase 5（長期）: Plan E — ハイブリッド統合

- 全ステージの統合
- 自動スケール判定
