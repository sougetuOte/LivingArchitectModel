## 1. Problem Statement

LAM プロジェクトは Milestone / Step / Wave / Task の 4 階層で作業を管理しているが（`.claude/rules/terminology.md` §1）、
現時点では作業状態の可視化手段が存在しない。進捗・ブロッカー・未完了タスクを把握するためには
SESSION_STATE.md や `docs/specs/` を手動で読む必要があり、全体像の即時把握が困難である。

本要件定義書は、LAM プロジェクトの作業状態を人間（ユーザー）が視覚的に把握できるダッシュボードを
定義する。ダッシュボードは「何が完了し、何が未完了で、何がブロックされているか」を一目で示す
単一の表示成果物として提供する。

---

## 2. Goals / Non-Goals

### Goals

- LAM プロジェクトの Milestone・Step・Wave・Task レベルの進捗状態を単一画面で確認できるようにする
- ユーザーが SESSION_STATE.md を開かずにプロジェクト全体の状態を把握できるようにする
- 既存のファイル群（SESSION_STATE.md、`docs/specs/`、`.claude/states/` 等）をデータソースとして活用し、
  別途データベースを構築しない
- 外部サービスへの接続なしにローカル環境でダッシュボードを表示できるようにする

### Non-Goals（非スコープ）

- ダッシュボードからの作業操作（タスク完了マーク、コメント投稿等）: 表示専用とする
- リアルタイムでの自動更新（更新間隔・プッシュ通知等）: Q4 で確定後に設計フェーズで判断
- チーム間のコラボレーション機能（共有・コメント・通知）
- 外部 CI/CD ツールとの統合
- MCP 連携を介したリモートアクセス: B-5 候補の将来段階（C 案）として future-candidates に記録
- モバイル対応・レスポンシブ最適化: PC ブラウザ表示を最低要件とする
- ダッシュボードのバージョン管理・履歴表示
- グラフ・チャートによる統計可視化（進捗率推移・バーンダウン等）: 将来候補

## 3. 機能要件（RFC 2119）

### §1: 階層表示

#### FR-1 ビュー構成（MUST）

ダッシュボードは以下のビュー単位を表示しなければならない。
各ビューの階層関係は `.claude/rules/terminology.md` §1 の用語階層（Project / Milestone / Step / Wave / Task）に準拠しなければならない。

| ビュー ID | 表示対象 | 必須表示情報 |
|-----------|---------|------------|
| V-1 | Project サマリー | Project 名・アクティブ Milestone 数・全体ステータス |
| V-2 | Milestone 一覧 | Milestone 名・現在の Step（属性列として表示）・状態（進行中 / 完了 / 未着手）|
| V-3 | Wave 一覧 | Wave 番号・Wave 内 Task 数・Wave 状態 |
| V-4 | Task 一覧 | Task ID・担当（AI 種別）・状態・完了判定 |

> **確定事項（PM 回答: 2026-06-20）**:
>
> - **V-1** は LAM 単一プロジェクト固定のトップサマリーページ（Q3=B 確定）。将来複数 Project 対応が必要になった場合はタブ追加で拡張する。
> - **V-2** の「現在の Step」は Milestone ビューの属性列として表示する。Step 独立ビュー（V-2.5 等）は設けない（Q2=B 確定）。
> - **V-5（個別実行ループ）は追加しない**。「個別実行ループ」は terminology.md の Task に統合済みであり、独立ビュー化しない（Q1=D 確定）。

#### FR-2 状態表示（MUST）

各階層の状態は以下の 4 値のいずれかで表示しなければならない:

| 状態値 | 意味 |
|-------|------|
| `not-started` | 未着手 |
| `in-progress` | 進行中 |
| `blocked` | ブロック中（前提条件未充足・未解決質問あり等）|
| `completed` | 完了 |

状態値が定義外の文字列になってはならない（MUST NOT）。

#### FR-3 ナビゲーション（SHOULD）

ビュー V-1 → V-2 → V-3 → V-4 の順で詳細へドリルダウンできることが望ましい。
ドリルダウンは同一ページ内のアンカーリンクまたはセクション展開で実現してよい（MAY）。

---

### §2: データソース読み込み

#### FR-4 データソース（MUST）

ダッシュボードは以下のファイルをデータソースとして読み込まなければならない。
ビュー単位でデータソースを固定する（Q5=D 確定: 優先順位ルールは設けない）:

| 対応ビュー | ファイル | 取得情報 |
|-----------|---------|---------|
| V-2, V-3 | `SESSION_STATE.md` | 現在の進行中タスク・未解決問題・次ステップ |
| V-2（Step 属性列） | `.claude/current-phase.md` | 現在のフェーズ（PLANNING / BUILDING / AUDITING）|
| V-4 | `docs/specs/<milestone>/tasks.md` | 各 Milestone のタスク一覧・完了チェック状況 |
| V-3（補完のみ）| `git log` 出力 | コミット履歴からの完了 Wave・Task の補完（コマンド出力であり、ファイルパスではない）|

> **確定事項（PM 回答: 2026-06-20・Q5=D）**: ビュー単位でデータソース固定。データソース間の競合は発生しない設計とする。
> データソース間で情報が矛盾する場合の解決ルールは設計フェーズで定義する。

#### FR-5 オフライン動作（MUST）

ダッシュボードはネットワーク接続なしにローカル環境で表示できなければならない。
外部 CDN・外部 API への依存を持ってはならない（MUST NOT）。

#### FR-6 ファイル読み込みエラーの扱い（MUST）

指定データソースファイルが存在しない場合またはパース不能な場合、
当該ファイルに依存するビューを「データなし」状態で表示し、
他のビューの表示を継続しなければならない。
エラーにより全ビューが白紙表示になってはならない（MUST NOT）。
`git log` の実行に失敗した場合は V-3 の補完情報なしで継続し、エラー終了してはならない（MUST NOT）。

### §3: 出力形式

#### FR-7 成果物形式（MUST）

ダッシュボードは単一の HTML ファイルとして出力しなければならない。

> **決定経緯**: 2026-06-18 セッション末の検討で「B=単一 HTML ダッシュボード」を第一推奨と位置付けた
> （SESSION_STATE.md §5「実装案 B 第一推奨」）。
> ただし最終的な実装方式（A=Mermaid 自動生成 / B=単一 HTML / C=MCP 連携 / ハイブリッド）の
> 選定は設計フェーズで再検証する。本要件は「単一 HTML ファイルとして出力すること」という
> 形式要件を定義するが、実現方法の選定は design フェーズの責任範囲とする。

#### FR-8 出力先（MUST）

HTML ファイルの出力先は `docs/artifacts/dashboard/` 以下としなければならない。
出力先ディレクトリが存在しない場合は自動作成しなければならない（MUST）。

> **確定事項（PM 回答: 2026-06-20・Q4=B）**: `/quick-save` 連動で頻繁に生成されるため、
> `docs/artifacts/dashboard/` は `.gitignore` に追加することを推奨する（設計フェーズで対応）。

#### FR-9a ビルドコマンド（SHOULD）

ダッシュボードの生成は、単一のコマンド実行（スクリプト呼び出し等）で完了できることが望ましい。

#### FR-9b ビルドコマンド実行環境（MUST）

ビルドコマンドは LAM プロジェクトの標準実行環境（Windows 11 / Git Bash / Python 3.x）で動作しなければならない。

---

### §4: スキル連携

#### FR-10 スキル `/build-dashboard`（SHOULD）

ダッシュボード生成を実行する Claude Code スキルを提供することが望ましい。
スキル名は **`/build-dashboard`** に確定（B-5 Wave 1 BUILDING / 2026-06-21 / UQ-3 解決）。
スキルは既存スキル（`/lam-orchestrate`、`/full-review`、`/quick-save` 等）と
命名規則・フロントマター書式を統一しなければならない（MUST、スキル実装時）。

> **確定経緯**: 旧仮称 `/visualize` から `/build-dashboard` に変更（動詞+名詞ケバブケース・既存スキル統一）。
> context7 MCP 利用不可のため `.claude/rules/upstream-first.md §3` フォールバック適用 →
> 既存スキル 4 件（`quick-save` / `project-status` / `lam-orchestrate` / `ship`）の
> `disable-model-invocation: true` 書式を権威的参照として採用。

#### FR-11 `/quick-save` との連携（SHOULD）

`/quick-save` 実行時にダッシュボードを自動更新する機能を提供することが望ましい（SHOULD）。
自動更新の実装方式は設計フェーズで確定する。

> **確定事項（PM 回答: 2026-06-20・Q4=B）**: MAY から SHOULD に格上げ。
> 自動更新により NFR-4（生成 30 秒以内）が `/quick-save` 実行時間に上乗せされる。
> AC は §7 AC-8 を参照。

## 4. 非機能要件

| ID | 要件 | 基準 |
|----|------|------|
| NFR-1 | HTML ファイルのサイズ | 単一ファイルとして 500 KB 未満（外部アセット参照なしの条件下）|
| NFR-2 | 表示完了時間 | ブラウザでファイルを開いてから 3 秒以内に全ビューが描画完了すること |
| NFR-3 | 対応ブラウザ | モダンブラウザ（Chrome 120+、Edge 120+）の最新安定版で表示できること |
| NFR-4 | ダッシュボード生成時間 | スクリプト実行開始から HTML ファイル生成完了まで 30 秒以内 |
| NFR-5 | 外部依存なし | 生成スクリプトは Python 3.x 標準ライブラリのみ使用するか、追加ライブラリの場合は `pyproject.toml` または `requirements.txt` で管理すること |
| NFR-6 | データソース欠如の耐障害性 | 全データソースファイルが存在しない場合でも HTML ファイル生成が完了すること（エラー終了禁止）|

> NFR-1〜6 の数値は初期 draft 値。設計フェーズで実測ベースに更新する。

## 5. 制約

- **プラットフォーム**: Windows 11 / Git Bash（パスはフォワードスラッシュ）。Bash ツール利用時の制約は `CLAUDE.md §Execution Environment` を参照。
- **外部サービス禁止**: ネットワーク接続を必要とする機能は本スコープに含まない（FR-5 に同じ）。
- **用語統一**: 全成果物の階層名・状態名は `.claude/rules/terminology.md` §1〜§2 の定義に従わなければならない（MUST）。「個別実行ループ」は terminology.md の Task に統合済みであり、本スコープでは独立定義しない（Q1=D 確定・2026-06-20）。
- **実装詳細の排除**: 具体的な HTML 構造・CSS クラス名・JavaScript フレームワーク選定は design フェーズの成果物に委ねる。

## 6. 用語

本節は b4-dashboard スコープで使用する用語の暫定定義を示す。
詳細は `docs/specs/b4-dashboard/glossary-draft.md` を参照。
terminology.md と矛盾する場合は terminology.md が優先する。

| 用語 | 定義 |
|------|------|
| ダッシュボード | LAM プロジェクトの作業状態を表示する単一 HTML 成果物 |
| ビュー | ダッシュボード内の表示単位（V-1〜V-4） |
| データソース | ダッシュボードが読み込む入力ファイル群。ビュー単位で固定（FR-4）|
| ビルド | データソースを読み込み HTML ファイルを生成する処理 |
| 状態値 | FR-2 で定義した `not-started` / `in-progress` / `blocked` / `completed` の 4 値 |

> Step は V-2 の属性列として表示する。「個別実行ループ」は terminology.md の Task に統合済みであり、本スコープに独立定義なし。

## 8. 解決済み質問（PM 回答: 2026-06-20）

本セクションの質問は、SESSION_STATE.md §5 に記述された「5 階層モデル
（プロジェクト一覧 / マイルストーン / Wave / Task / 個別実行ループ）」と
`.claude/rules/terminology.md`（Project / Milestone / Step / Wave / Task の 4 + Phase 補助）の
**不整合** に起因していた。全質問が PM 回答により 2026-06-20 に解決済みとなり、設計フェーズへの着手が可能となった。

### Q1: 「個別実行ループ」の定義 — **解決済み（PM 回答: 2026-06-20）**

**回答**: D（別途定義は行わない）

**判断理由**: 「個別実行ループ」は terminology.md の Task に統合する。独立した階層・ビューは設けない。
SESSION_STATE.md の旧記述（5 階層モデル）は terminology.md §5 移行猶予条項により温存する（修正不要）。

**反映箇所**: FR-1（V-5 追加なし）、FR-1 注記、§5 制約、§6 用語、§9 将来候補

---

### Q2: Step（PLANNING / BUILDING / AUDITING）の可視化方針 — **解決済み（PM 回答: 2026-06-20）**

**回答**: B

**判断理由**: Step は Milestone ビュー（V-2）の「現在の Step」属性列として表示する。独立ビュー（V-2.5 等）は設けない。

**反映箇所**: FR-1 テーブル（V-2 列「現在の Step」明示）

### Q3: 「プロジェクト一覧」の意味 — **解決済み（PM 回答: 2026-06-20）**

将来複数 Project 対応が必要になった場合はタブ追加で拡張する（設計変更不要）。

**反映箇所**: FR-1（V-1 表示対象を「Project サマリー」に変更）、FR-1 注記

---

### Q4: ダッシュボードの更新トリガー — **解決済み（PM 回答: 2026-06-20）**

**回答**: B（`/quick-save` 連動）

**反映箇所**: FR-11（MAY → SHOULD）、FR-8 注記（gitignore 推奨確定）、AC-1 更新、AC-8 追加

---

### Q5: データソースの優先順位と競合解決 — **解決済み（PM 回答: 2026-06-20）**
