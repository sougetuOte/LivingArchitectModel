## コンテキスト

### 背景: Fable 5 のクレジット従量移行

Fable 5 は 2026-07-07 まで Pro/Max weekly limit の 50% 枠で利用可能だが、以降はクレジット従量に移行する。
API 単価は入力 $10/MTok・出力 $50/MTok であり、Opus 4.8（$5/$25）の約 2 倍。「常時 L1 として Fable を使う」
運用は非経済的になる。一方で Fable の強みは **タスクが長く複雑になるほど差が開く** 領域にあり、
spec/design 初期の設計軸確定や真の行き詰まり局面では Opus 単独より高い価値を持つ。

この非対称性（常駐は高コスト・スポット起用は高価値）から、Fable を「常駐 daemon」ではなく
「儀式的に召喚する上位オラクル」として扱う HGA（HolyGuardianAngel）型アプローチが
2026-07-01 に討議・起草された（`docs/artifacts/hga-approach-2026-07-01.md`）。

### 確定事実（2026-07-02 公式裏取り済み — 討議録の未解決前提を解消）

討議録 §5.1 day-1 実測チェックリストの一部は、本 ADR 起票時点で以下の通り解決済みと確認できた。

| # | 項目 | 状況 |
|---|------|------|
| 1 | プロンプトキャッシュがクレジット従量 Fable に効くか | **解決済み**。キャッシュはクレジット従量にも適用される（read $1/MTok = 0.1x / 5m write 1.25x / 1h write 2x） |
| 3 | Fable が Claude Code で web/tool を使えるか | **解決済み**。実セッションでフルアクセス可を確認済み |
| 4 | routing/Opus フォールバックの発火頻度と可視性 | **可視であることが判明**。safety classifier 発火時は Opus 4.8 で再実行 + transcript に通知表示 + `/model fable` まで Opus 継続。発火領域は攻撃的セキュリティ・生物/化学等であり設計討議ではまず発火しない → 監視で対応可 |
| 2 | 自己修復ループの往復回数を実際に抑えられるか | **未解決**。運用開始後の実測が必要（本 ADR の day-1 実測項目として継続） |
| 5（新規） | ブリーフの実効入力トークン | **未解決**。Claude Code セッションは system prompt + CLAUDE.md + rules がブリーフ以前に載るため、15-20k ブリーフの実効入力は 40-60k になりうる |

補足事実: 支出上限は Pro/Max で月次キャップを自設定でき、到達時は自動 pause する（討議録が前提とした
固定 $100 上限の公式記述はない）。また Claude Code の Agent ツールには `model: "fable"` 指定が存在し、
Opus L1 セッションから Fable を subagent 召喚する経路が実在する（使い捨て・leaf・ブリーフ渡しという
HGA 型の構造に一致）。実効入力量は実測項目として残る。

### 問題

1. 「常時 L1」は非経済だが、Fable を全く使わないと adequacy（設計の十分性）軸で劣後するリスクがある
   （Opus 単独の天井に起因する crux 見逃し）
2. Fable 召喚には往復コストの温床（自己修復ループ）と鮮度ギャップ（currency）という構造的な穴がある
3. 討議録 §9 で Fable 自身に上申した 5 争点（E: 枠棄却／A: ブリーフ adequacy の床／B: キャッシュ経済／
   C: 随時協議のパラドクス／D: 価格状態への over-fit）への回答を設計に反映する必要がある
4. LAM 既存の 3.5 層委譲モデル（`CLAUDE.md` §作業体制）との整合を取る必要がある

### 制約条件

- LAM Hierarchy of Truth: User Intent が最上位（ユーザー承認 2026-07-02 と整合）
- 権限等級: **PM 級**（`docs/adr/` の新規作成 + `.claude/rules/` の新規作成 + `CLAUDE.md` の変更）
- upstream-first 原則: Fable の課金体系・safety routing・Agent ツール `model` 指定は公式裏取り済みの
  確定事実として扱い、未確認事項は明示的に「要実測」と記録する

## 検討した選択肢

### 案 1: 却下 — Fable 常駐 L1

**概要**: 従来通り Fable を L1（統括）に据え、常時利用する。

**却下理由**:
- コスト軸で劣後。$10/$50 per MTok は Opus の約 2 倍であり、日常的な判断・査定作業まで
  高単価モデルに載せるのは非経済
- 討議録 §1 の「常時 L1 は非経済」という前提を覆す材料がない

### 案 2: 却下 — Fable 不使用（Opus 単独運用）

**概要**: Fable を一切使わず、既存の Opus/Sonnet/Haiku 体制のみで運用する。

**却下理由**:
- adequacy 軸で劣後。MELCHIOR / BALTHASAR / CASPAR は同一モデル（Opus）の別ペルソナであり
  盲点が相関する。難所ほど三者は偽の合意に収束し、割れるべき所で割れない
  （討議録 §3「coherent ≠ adequate」）
- 天井起因の crux 見逃しは同一天井内の自己監査では解消できない構造的限界

### 案 3: 却下 — branch モードを既定にする

**概要**: Fable に Sonnet を直接ぶら下げる（Fable=branch）構成を既定運用にする。

**却下理由**:
- **階層反転**: 入れ子は能力降順前提の設計だが、branch では L2 の Fable が L1 の Opus より賢く、
  orchestration コストを最賢ノードに払わせることになる
- **正本/TTL 喪失**: 蓄積 retrieval が Fable の揮発 context に載り、漏れたら全消失。
  「Fable は使い捨て」原則（コア原則 5）が崩れる
- **コスト発散**: Fable の spawn ループ = $50/MTok 出力 × 多ターンであり、これまで避けてきた
  発散そのものを再現する
- 討議録 §4.1 により branch は「tight な適応探索」のみの例外・別予算枠と位置づけ済み

### 案 4: 採用 — HGA 型（Opus 媒体・圧縮ブリーフ・スポット召喚・ステートレス化）

**概要**: 討議録の HGA 型アプローチを採用する。Opus を媒体・正本保持者とし、圧縮ブリーフで
Fable をスポット召喚、Fable セッションは使い捨て（leaf）とする。§9 上申への Fable 回答
（下記「決定」参照）を設計に統合する。

**採用理由**:
- コスト最小 × 恩恵最大の交点を突く（低頻度・高 stakes 局面にのみ高単価モデルを起用）
- TTL 問題が正本=Opus 側保持により線形被害に留まる（指数被害を回避）
- LAM 既存の 3.5 層委譲モデルと整合（L1=Opus が判断・媒体を担い、Fable は特別スポットとして
  その外側に位置づけられる）
- Fable 自身の §9 回答により、設計の穴（ブリーフ adequacy・往復コスト・ステートレス化・
  over-fit 対応）に具体的な緩和策が与えられている

## 決定

採用案 4（HGA 型）を、討議録 §9 上申に対する Fable 5 の回答（2026-07-02）を反映した形で実装する。

### コア原則（8 か条 — 討議録 §2 の 7 か条 + ステートレス化を第 8 条として追加）

討議録 §2 の 7 か条（Opus が媒体／召喚前の禊／stakes×novelty で召喚／出力は分岐点と根拠のみ／
正本は Opus・Fable は使い捨て／currency は push／Fable は既定で leaf）に加え、
争点 C への回答（下記）により以下を第 8 条とする。

| # | 原則 |
|---|------|
| 8 | **完全ステートレス化**: 人間との協議は Opus 側のみで行い、Fable には確定争点のみをステートレス再召喚で渡す。人間の判断待ちで Fable セッションを開けたまま保持しない |

### 争点 E（枠棄却）の決定記録

**枠棄却は 2026-07-02 に検討済み・優越案なし。** 討議録 §9 条件 (3) に従い、HGA 型を上回る
アプローチは提示されなかった。この結論を決定として記録し、**条件変化（価格レジーム変化・
サブスク復帰・Fable 能力の大幅変化）まで再論しない**（討議録 §9 スコープ規定「一度回答を
得たら決定として記録し、条件変化まで再論しない」の初回適用）。

### 争点 A〜D への回答の反映

| 争点 | 採択された修正 |
|------|---------------|
| **A**（ブリーフ adequacy の床） | **2 段召喚**: 無条件召喚ゾーンでは、フルブリーフの前に 2-3k の問題スケッチのみ渡し「crux の所在と必要資料」を先に Fable に問い、その指示で Opus がブリーフを編む。**索引 push**: ブリーフに「Opus が要約対象から外した原資料の目次・ファイル名一覧」を数百トークンで同梱（必須部品）。残る床（索引にも載らない概念的 crux）は既約として受容する |
| **B**（キャッシュ経済） | キャッシュ適用は確認済み（上記コンテキスト参照）だが「割引」と位置づけ、頼みにしない。コスト防御の本丸は圧縮に置く。**隠れコスト**として、Claude Code セッションは system prompt + CLAUDE.md + rules がブリーフ以前に載るため、15-20k ブリーフの実効入力は 40-60k になりうる点を day-1 実測に追加する |
| **C**（随時協議のパラドクス） | **完全ステートレス化**（上記コア原則第 8 条）により解消。人間との協議は Opus 側のみで完結させ、Fable への再召喚は確定争点のみをステートレスに渡す。1 召喚内の資料要求往復は原則 1 回以内。TTL は「守る対象」から「無視する対象」に変わり、討議録 §5 の緊張（バッチ化 vs 随時協議）は消滅する |
| **D**（価格状態への over-fit） | 不変核と state-dependent 部分を分離（下記「D（over-fit）対応」参照） |

### 別予算 2 枠（envelope 外）

討議録 §4.1 の branch モード例外に加え、以下を envelope（月 $40-80）の外に置く別予算 2 枠とする。
計上ラベルを分離することで、コスト暴発源を切り分け可能にする。

| 枠 | 内容 |
|----|------|
| 対話モード召喚 | 真の行き詰まり時のみ、人間を含めた協議を行う召喚 |
| branch モード | 討議録 §4.1 例外（tight な適応探索のみ・稀） |

### 漏れ回収の 3 経路

運用者が混同しやすいため、以下 3 経路を明確に区別する（`.claude/rules/hga-summoning.md` に
運用規律として反映）。

| 経路 | 内容 | 回収手段 |
|------|------|---------|
| 見える漏れ | 索引 push で存在が示された資料 | Fable が pull 要求可 |
| 鮮度（currency） | 知識の鮮度ギャップ | Fable からは要求不能・push のみ |
| 概念的 crux | ブリーフ・索引双方に現れない天井起因の不足 | 既約・2 段召喚で縮小するが完全解消はしない |

### 召喚記録

全召喚を `docs/artifacts/hga-summon-log.md` に追記する。これは争点 E の再論禁止規律と
envelope 監視の実行基盤となる。

### 実装経路

Claude Code の Agent ツールに `model: "fable"` 指定が存在し、Opus L1 セッションから Fable を
subagent として召喚する経路が実在する（使い捨て・leaf・ブリーフ渡しという HGA 型の構造に一致）。
実効入力量（B への回答の隠れコスト）は day-1 実測項目とする。

### day-1 実測（推奨タイミング）

討議録 §5.1 のうち未解決の項目（往復回数の抑制・実効入力トークン量）は、**2026-07-07 までの
週次 50% 枠内**で実測を済ませることを推奨する。従量課金移行前に前提を確定させるため。

## D（over-fit）対応: 不変核と state-dependent の切り分け

価格レジームが変わっても価値が残る不変核と、現行の従量課金前提に紐づく state-dependent な
足場を分離する。

### 不変核（価格レジームによらず維持）

1. **選別・圧縮ロジック**（何を難所と見て何を Fable に渡すか。LAM 固有資産）
2. **currency push**（鮮度ギャップは pull 不可という構造的事実。課金体系に依存しない）
3. **stakes × novelty 召喚ゲート**（低頻度・高 stakes 局面にのみ上位知性を起用する設計思想）
4. **正本の媒体側保持**（Opus が context の真実を持ち、Fable は使い捨てという役割分担）

### state-dependent（従量課金前提に紐づく — サブスク復帰時は撤去対象）

- 討議録 §5 のコスト管理全体（TTL 熱管理・envelope 額・レンジ試算）
- コア原則第 4 条（出力は分岐点と根拠のみに縛る、という**出力縛り**）
- leaf 既定のコスト論拠（ただし階層反転論拠は残るため、leaf 自体の構成は残り、
  「コストを理由にした leaf 強制」のみが撤去対象）

サブスク復帰時は、state-dependent 部分のみを撤去する。具体的な撤去手順は復帰が実現した
時点で別途 ADR として起票する（軽微な運用調整のため本 ADR の再起票は不要な場合は
`.claude/rules/hga-summoning.md` の改訂で対応する）。

## 結果（Consequences）

### Positive

- コスト最小 × 恩恵最大の交点を突く設計（低頻度・高 stakes 局面にのみ Fable を起用）
- 完全ステートレス化により TTL 問題が構造的に消滅（討議録 §5 の未解決の緊張を解消）
- LAM 既存の 3.5 層委譲モデルと整合（`CLAUDE.md` §作業体制の読み替え規定で吸収）
- 2 段召喚 + 索引 push によりブリーフ adequacy の床が縮小（争点 A への具体的緩和）
- 争点 E の決定記録により、毎召喚での枠論議という自壊シナリオを回避

### Negative

- Fable × 人間の多ターン共同推論は別予算の対話モード枠のみに限定され、日常運用からは排除される
- 概念的 crux（索引にも載らない天井起因の不足）は既約な床として残り、完全には解消されない
- 実効入力トークンの隠れコスト（B への回答）が確定するまでは、月 $40-80 envelope の精度に
  不確実性が残る

### Neutral

- day-1 実測 5 項目のうち 3 項目（キャッシュ適用・web/tool アクセス・routing 可視性）は
  ADR 起票時点で解決済み。残り 2 項目（往復抑制・実効入力トークン）が前提として残る。
  これらが崩れた場合、コスト概算・往復設計の再検討が必要になるが、コア原則・不変核には
  影響しない設計としている
