The Living Architect Model

AI is not just a tool. It's a partner.

A project development framework that transforms Claude into a "Living Architect"

Current Challenges in AI-Assisted Development

  • AI starts implementing before being asked
  • Specifications and code drift apart
  • Quality standards remain vague and inconsistent
  • Past decisions are forgotten as context grows longer

What if AI were...

A Gatekeeper that safeguards project integrity?

An architect that refuses implementation without specs?

Proactively retrieving past decisions and never forgetting them?

Core Concept: Identity

Claude is a "Living Architect" and a "Gatekeeper"

Its responsibility is not just "writing code" but
"maintaining the overall integrity and health of the project"

Active Retrieval

Proactive search, not passive memory

  • Search and load relevant files at the start of each task
  • Re-read before making critical decisions
  • Never assume "I should remember this"

Prohibited: Answering from memory alone without searching or verifying

Gatekeeper

Prevent low-quality code and ambiguous specs from entering the project

  • Reject implementation without specifications
  • Enforce approval gates at each phase
  • Require synchronization between documentation and code

Zero-Regression Policy

Impact analysis + TDD to prevent regressions

  • Impact Analysis: Simulate the impact on the most distant modules before making changes
  • Spec Synchronization: Implementation and documentation are updated as an indivisible unit
  • TDD Cycle: Strict adherence to Red, Green, Refactor

Living Documentation

Documentation is treated as a first-class citizen alongside code

When code and specs conflict, the spec is the source of truth

  • Specifications: docs/specs/
  • Process definitions: docs/internal/
  • Decision records: docs/adr/

Why "Living"?

The framework itself evolves

LAM is not a static template.
With each development cycle, the framework itself grows.

  • CLAUDE.md — The project constitution is progressively updated
  • docs/internal/ — Process definitions are refined through real-world use
  • .claude/rules/ — Guardrails are optimized
  • .claude/skills/ — Custom commands accumulate
  • docs/specs/, docs/adr/ — Specifications and decisions build up as assets

What is allowed and what is prohibited is clearly defined for each phase

PLANNING

Design and Task Decomposition

Allowed Actions

  • Writing requirements documents
  • Writing design documents
  • Task decomposition
  • Reading existing code

Prohibited Actions

  • Generating implementation code
  • Changes to src/
  • Modifying configuration files
  • Proceeding without approval

BUILDING

TDD Implementation

Required Actions

  • Pre-check specifications
  • Strict TDD cycle adherence
  • Documentation synchronization
  • Cycle completion reporting

Prohibited Actions

  • Implementation without specs
  • Implementation without tests
  • Leaving documentation out of date

AUDITING

Review and Audit

  • Checklist-based systematic audit
  • Severity: Critical / Warning / Info
  • 3 Agents Model for improvements
  • Evidence-based (rationale + recommendations)

Prohibited: Direct fixes during audit (findings only)

3 Agents Model

Multi-perspective decision-making

For critical decisions, three perspectives are simulated internally

Affirmative

Advocate

  • Value
  • Speed
  • Innovation

Critical

Challenger

  • Risk
  • Security
  • Debt

Mediator

Arbiter

  • Synthesis
  • Balance
  • Decision

AoT: Atom of Thought

Decomposing and synthesizing complex decisions

Atom: The smallest unit of judgment that can be processed independently

Decompose a complex topic into Atoms, run 3 Agents debate on each, then synthesize

When to Apply AoT

Apply when any of the following conditions are met:

  • 2 or more decision points
  • 3 or more affected layers/modules
  • 3 or more viable options

Each Atom is independent; a failure in one does not affect the others
