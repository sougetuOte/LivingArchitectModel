Use Case:
Growing Your Project

The framework itself grows
— that's what "Living" means

Tracing how LAM and project assets evolve
across multiple development cycles

Scenario

Goal: Develop a new web app using LAM

  • 1st User Authentication
  • 2nd Product Management
  • 3rd Order Processing

Key focus: Not the features themselves, but the accumulation of project assets and the evolution of the framework

Phase 0: Initial Setup

What you deploy

  • CLAUDE.md (template)
  • .claude/rules/ (base rules)
  • .claude/skills/ (standard commands)
  • docs/internal/ (general processes)

Still empty

  • docs/specs/ (specifications)
  • docs/adr/ (decision records)
  • docs/tasks/ (task definitions)

At this point, LAM is still just a "generic template"

1st Cycle Complete: What Was Born

Project Assets

  • 1 specification
  • 1 ADR
  • Test suite
  • Implementation code

Insights Gained

  • JWT operational policy established
  • Security requirement patterns
  • Testing strategy standards
  • Code review perspectives

Problem: These insights remain "tacit knowledge." How do we leverage them in the next cycle?

Framework Update 1

Feeding tacit knowledge back

  • CLAUDE.md — Security policies & tech stack criteria added
  • docs/internal/ — Auth design patterns added
  • .claude/rules/ — Project-specific security rules

LAM transforms from a template into a "project-tailored framework"

2nd Product Management PLANNING

You

/planning

Living Architect

Difference from 1st: Past specs, ADRs, and rules are automatically referenced, promoting consistent design

2nd Cycle Complete: What Else Has Grown

Accumulated Assets

  • 2 specifications (cross-referenced)
  • 3 ADRs
  • Test infrastructure reuse

Framework Update 2

  • CLAUDE.md — DB design conventions added
  • skills/ — CRUD generation skill added
  • docs/internal/ — Testing strategy revised

CLAUDE.md has evolved from a "template" into a "crystallization of project wisdom"

3rd Order Processing

Living Architect

Verified consistency with auth (spec/001) and products (spec/002).
Following DB conventions from ADR-003, leveraging CRUD skills for efficient task decomposition.

Assets leveraged
  • 2 specs → Consistent API design reference
  • 3 ADRs → Building on past decisions
  • CRUD skill → Routine task automation
  • Security rules → Auto-applied

1st Cycle's hard decisions are now automatically guided in the 3rd

The Compound Effect of Growth

1st

Building the Foundation

  • Everything from scratch
  • Using templates
  • Takes time

2nd

Leveraging Assets

  • Referencing existing specs/ADRs
  • Rules auto-applied
  • Efficiency improves

3rd

Automated Guidance

  • Rich assets enable high precision
  • Skills automate tasks
  • Significantly accelerated

Evolution of CLAUDE.md

The project constitution is alive

Right after adoption

  • Identity (template)
  • Hierarchy of Truth (generic)
  • Core Principles (generic)
  • Execution Modes (basic 3)

After 3 Cycles

  • Identity + Project-specific duties
  • Truth + Tech stack criteria
  • Principles + Security & DB rules
  • Modes + Custom workflows

CLAUDE.md grows = AI partner's judgment refines

With LAM vs Without

Without LAM

  • Design from scratch every time
  • Past decisions are forgotten
  • Documentation decays
  • AI quality stays flat

With LAM

  • Assets accumulate and get reused
  • Proactively searches past records
  • Documentation stays in sync
  • AI partner gets smarter

The more you use it, the wider the gap. That's the compound effect of "Living"

The True Value of "Living"

  • CLAUDE.md doesn't stay a template. It becomes a crystallization of project wisdom
  • docs/internal/ isn't write-and-forget. It gets refined through real-world use
  • rules/ and skills/ keep growing. AI behavior becomes optimized
  • specs/ and adr/ keep being referenced. Decisions are never forgotten

The further the project progresses, the smarter your AI partner becomes.
That is the Living Architect Model.
